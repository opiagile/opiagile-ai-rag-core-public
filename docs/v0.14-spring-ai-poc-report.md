# Relatório v0.14 — Prova De Conceito Spring AI

## Resumo Executivo

A v0.14 iniciou a migração guiada por Spring AI pelo menor ponto de risco: embeddings. O provider manual OpenAI continua sendo o padrão, e a nova integração Spring AI entra apenas por configuração explícita.

O objetivo foi validar que o core pode usar `EmbeddingModel` do Spring AI sem quebrar:

- modo local sem chave;
- fallback textual;
- contrato `EmbeddingProvider`;
- recuperação pgvector existente;
- testes atuais.

## O Que Foi Implementado

- Adicionado BOM Spring AI `1.1.8`, compatível com a linha Spring Boot 3.5.x usada pelo projeto na v0.14. Na v0.15, o baseline foi atualizado para Java 25 LTS, Spring Boot 4.1.x e Spring AI 2.0.x.
- Adicionado `spring-ai-starter-model-openai`.
- Criado `SpringAiEmbeddingProvider`.
- Criado `FallbackEmbeddingProvider` para impedir falha de startup quando a configuração Spring AI estiver incompleta.
- Ajustado `SpringAiEmbeddingProvider` para resolver `EmbeddingModel` via `ObjectProvider`, evitando dependência de ordem entre component scan e auto-configuração Spring AI.
- Ajustado `OpenAiEmbeddingProvider` para continuar padrão com `OPENAI_EMBEDDINGS_PROVIDER=manual`.
- Adicionadas configurações:
  - `OPENAI_EMBEDDINGS_PROVIDER=manual|spring-ai`;
  - `SPRING_AI_MODEL_CHAT=none`;
  - `SPRING_AI_MODEL_EMBEDDING=none|openai`.
- Atualizados `.env.example`, Compose Oracle e documentação.
- Adicionados testes de provider Spring AI.
- Criado script de validação real `scripts/validate-spring-ai-embeddings.py`.
- Criado workflow manual `.github/workflows/validar-spring-ai-embeddings.yml`.

## Como Ativar A POC

```text
OPENAI_EMBEDDINGS_ENABLED=true
OPENAI_EMBEDDINGS_PROVIDER=spring-ai
SPRING_AI_MODEL_EMBEDDING=openai
OPENAI_API_KEY=<valor seguro fora do Git>
OPENAI_EMBEDDING_MODEL=text-embedding-3-small
OPENAI_EMBEDDING_DIMENSIONS=1536
```

O chat Spring AI continua desabilitado por padrão:

```text
SPRING_AI_MODEL_CHAT=none
```

Isso evita criar beans de chat Spring AI antes da avaliação específica do `ChatModelProvider`.

## Decisão Técnica

O provider manual segue como padrão porque:

- já está validado em produção de demonstração;
- usa OpenAI Responses API;
- expõe metadados específicos usados em trace;
- permite controle fino de `reasoning.effort` e `text.verbosity`.

O provider Spring AI foi adicionado apenas para embeddings, onde o contrato é menor e a comparação é mais segura.

## Segurança

- Nenhuma chave foi adicionada.
- `.env` não foi alterado.
- O provider Spring AI não registra `OPENAI_API_KEY`.
- Em falha de provider, o resultado volta vazio e o RAG usa fallback textual.
- Se o `EmbeddingModel` do Spring AI não estiver disponível, a aplicação continua subindo e o RAG preserva fallback textual.

## Validação Real Com GitHub Actions

Workflow manual:

```text
Validar embeddings Spring AI
```

Arquivo:

```text
.github/workflows/validar-spring-ai-embeddings.yml
```

O workflow usa `OPENAI_API_KEY` via GitHub Secret, sobe PostgreSQL/pgvector, inicia a API com:

```text
CHAT_RESPONSE_MODE=DEMO
LLM_PROVIDER=DEMO
OPENAI_EMBEDDINGS_ENABLED=true
OPENAI_EMBEDDINGS_PROVIDER=spring-ai
SPRING_AI_MODEL_CHAT=none
SPRING_AI_MODEL_EMBEDDING=openai
TOOLS_PLANNER_LLM_ENABLED=false
```

Em seguida, o script:

1. valida `/actuator/health`;
2. valida `/api/providers/status`;
3. ingere `samples/clinica/faq.txt`;
4. confirma `embeddingProvider=spring-ai` no upload;
5. chama `/api/chat`;
6. consulta o trace da conversa;
7. exige provider de recuperação `pgvector-spring-ai`.

Essa validação não usa LLM de chat. Ela mede especificamente embeddings Spring AI, persistência dos vetores e recuperação pgvector.

Resultado validado em GitHub Actions:

- Run: `28406561642`;
- Status: `success`;
- Data: `2026-06-29`;
- `embeddingProvider`: `spring-ai`;
- `chunkCount`: `3`;
- `sourcesCount`: `3`;
- provider de recuperação no trace: `pgvector-spring-ai`;
- `GET /api/providers/status`: embeddings `OPERACIONAL`, estratégia `PGVECTOR_COM_FALLBACK_TEXTUAL`, sem warnings.

Validações complementares recomendadas antes de trocar o padrão:

1. Ingerir documento com `OPENAI_EMBEDDINGS_PROVIDER=manual`.
2. Ingerir o mesmo documento com `OPENAI_EMBEDDINGS_PROVIDER=spring-ai`.
3. Comparar dimensão, quantidade de chunks com embedding, latência e recuperação.
4. Confirmar que `GET /api/providers/status` mostra provider e fallback esperados.
5. Executar o workflow manual `Validar embeddings Spring AI` no GitHub Actions.
