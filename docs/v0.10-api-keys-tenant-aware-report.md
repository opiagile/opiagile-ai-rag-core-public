# Relatório v0.10 — API Keys Tenant-aware

## Resumo Executivo

A sprint v0.10 adicionou uma camada de autorização server-to-server para o `opiagile-ai-rag-core`.

O core agora suporta API keys vinculadas a tenant/workspace, com escopos por operação e rate limit por client. Isso reduz o risco de depender de headers livres `X-Tenant-Id` e `X-Workspace-Id` em ambientes expostos.

O modo local legado foi preservado para desenvolvimento e compatibilidade.

## O Que Foi Implementado

- Tabela `api_clients` via Flyway.
- Hash SHA-256 de API key, sem armazenar chave real em texto puro.
- `ApiClientAuthenticationFilter`.
- `ApiClientRepository`.
- `ApiClientContextHolder`.
- `ApiSecurityProperties`.
- Priorização do tenant/workspace autenticado no `TenantContextResolver`.
- Compatibilidade com `X-OPIAGILE-API-KEY` e `Authorization: Bearer`.
- Escopos por endpoint.
- Rate limit em memória por API client.
- Execução de tools autorizada por `tools:execute` ou token administrativo legado.
- Script `scripts/generate-api-client-key.sh` para gerar chave e SQL de cadastro.

## Escopos

| Escopo | Operação |
| --- | --- |
| `chat:write` | `POST /api/chat` |
| `documents:read` | `GET /api/documents*` |
| `documents:upload` | `POST /api/documents/upload` |
| `conversations:read` | `GET /api/conversations*` |
| `observability:read` | `GET /api/observability*` |
| `handoffs:read` | `GET /api/handoffs*` |
| `handoffs:write` | `PATCH /api/handoffs*` |
| `tools:read` | `GET /api/tools` |
| `tools:execute` | `POST /api/tools/{slug}/sql/query` |
| `workspaces:read` | `GET /api/workspaces` |

## Configuração

```text
API_SECURITY_ENABLED=true
API_SECURITY_REQUIRE_API_KEY=false
API_SECURITY_DEFAULT_RATE_LIMIT_PER_MINUTE=60
```

Para ambiente exposto ou multiempresa, a recomendação é:

```text
API_SECURITY_REQUIRE_API_KEY=true
```

## Como Gerar Uma Chave

```bash
scripts/generate-api-client-key.sh demo clinica-demo "Gateway Demo" "chat:write,documents:read,documents:upload,conversations:read,observability:read,workspaces:read" 60
```

O script imprime:

- a API key, que deve ser copiada e guardada em secret seguro;
- o SQL de cadastro com o hash SHA-256.

A chave real não deve ser commitada, documentada nem armazenada em texto puro.

## Compatibilidade

O modelo legado continua funcionando quando `API_SECURITY_REQUIRE_API_KEY=false`:

- headers `X-Tenant-Id` e `X-Workspace-Id`;
- `DEMO_ACCESS_TOKEN`;
- `DEMO_ADMIN_TOKEN`;
- `SITE_CHAT_API_KEY`;
- segurança própria do piloto WhatsApp.

Quando uma API key válida é enviada, o tenant/workspace da credencial prevalece sobre headers livres.

## Limitações

- O rate limit é em memória; múltiplas instâncias exigiriam Redis ou mecanismo externo.
- Ainda não há login de usuário final, JWT/OIDC ou RBAC de painel.
- API keys são voltadas a gateways server-to-server, não a usuários finais no navegador.
- Conversas, handoffs e observabilidade foram protegidos por escopo, mas a evolução ideal é filtrar todos esses repositórios também por tenant/workspace quando houver painel administrativo multiusuário.
