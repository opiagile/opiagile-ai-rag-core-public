# Relatório v0.4 - LLM Conversacional E Demo UI

## Resumo Executivo

A Sprint v0.4 corrigiu a principal limitação percebida nos testes manuais: o chat anterior era determinístico e não gerava respostas com LLM no fluxo principal.

Agora o backend possui geração conversacional opcional com LLM via OpenAI Responses API, mantendo fallback local sem chave externa. A Demo UI também foi redesenhada para parecer uma experiência de atendimento, com conversa em balões, painel de contexto, fontes, handoff e trace.

O projeto continua não sendo ambiente produtivo, demo pública aberta, integração real com WhatsApp ou RAG vetorial completo.

## Estado Anterior

Antes desta sprint:

- `ChatService` montava a resposta em `buildAnswer(...)`.
- A resposta usava frases fixas e o melhor chunk recuperado.
- Não havia prompt conversacional no fluxo principal.
- `LlmIntentClassifier` existia como extensão, mas não gerava resposta de atendimento.
- A Demo UI funcionava, mas tinha aparência mais técnica, próxima de uma tela de teste de API.

## O Que Mudou

Backend:

- Criada abstração `ChatModelProvider`.
- Criado `DemoChatModelProvider` para manter o modo local sem chave externa.
- Criado `OpenAiChatModelProvider` para geração opcional por OpenAI.
- Criado `RoutingChatModelProvider` para escolher LLM ou fallback local.
- Criado `RagAnswerPromptBuilder` com prompt de atendimento em português.
- `ChatService` passou a gerar resposta usando histórico, intenção, lead status, handoff e fontes recuperadas.
- `ChatResponse` passou a retornar metadados opcionais: `responseMode`, `llmProvider`, `model`, `fallbackReason`, `llmLatencyMs`, `promptTokens`, `completionTokens` e `totalTokens`.
- Observabilidade passou a registrar modo de resposta, provider LLM, modelo, latência LLM e tokens quando disponíveis.

Demo UI:

- Interface reorganizada como inbox de atendimento.
- Chat central em balões de conversa.
- Painel lateral com contexto, modo DEMO/LLM, fontes, handoff, status da API e trace.
- Badges para API, modo de resposta, RAG local e WhatsApp mock.
- Mensagens mais claras para visitante leigo.

## Como Rodar Sem LLM

Use o modo padrão:

```text
CHAT_RESPONSE_MODE=DEMO
LLM_PROVIDER=DEMO
OPENAI_API_KEY=
```

Neste modo, o projeto continua rodando sem segredos reais e responde com fallback local determinístico.

## Como Ativar LLM

Configure no `.env` local, sem versionar segredo:

```text
CHAT_RESPONSE_MODE=LLM
LLM_PROVIDER=OPENAI
OPENAI_API_KEY=<sua-chave-local>
OPENAI_CHAT_MODEL=gpt-5-mini
OPENAI_MAX_OUTPUT_TOKENS=600
OPENAI_TIMEOUT_SECONDS=30
```

Com essas variáveis, o chat usa a OpenAI Responses API para gerar resposta natural com base no histórico e nas fontes recuperadas. Se a chamada falhar ou a chave estiver ausente, o fallback local é usado e `fallbackReason` é preenchido.

## Limitações Mantidas

- Embeddings reais ainda não são o fluxo principal.
- Recuperação atual continua textual/local no modo sem chave externa.
- WhatsApp real ainda não foi implementado.
- Demo pública HTTPS ainda está pendente.
- Não há autenticação nem rate limit para exposição pública.
- Uso de LLM depende de chave, conectividade e custo do provedor.

## Próxima Sprint Recomendada

v0.5 - WhatsApp Cloud API Piloto Controlado.

A v0.5 deve vir depois de validar a v0.4 com uma chave real, porque o WhatsApp deve expor uma experiência conversacional mais natural para testers externos.
