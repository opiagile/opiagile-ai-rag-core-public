# Relatório v0.12 — Portal De Desenvolvedores E OpenAPI

## Resumo Executivo

A v0.12 adicionou uma experiência de desenvolvedor para empresas e parceiros testarem o contrato HTTP do `opiagile-ai-rag-core`.

O objetivo é demonstrar capacidade técnica e facilitar integrações sem transformar o core em frontend de produto. A documentação interativa usa OpenAPI/Swagger, mas com portal visual e tema Opiagile.

## O Que Foi Implementado

- Dependência `springdoc-openapi-starter-webmvc-ui`.
- Configuração OpenAPI com:
  - título e descrição comercial;
  - contato Opiagile;
  - licença MIT;
  - servers local e demonstração;
  - autenticação por `X-OPIAGILE-API-KEY`.
- Grupo OpenAPI `rag-core`.
- Portal visual:

```text
GET /developers
GET /developers/
```

- Console Swagger com tema customizado:

```text
GET /developers/console
GET /developers/swagger-ui/index.html
```

- Especificação JSON:

```text
GET /v3/api-docs/rag-core
```

- Ajuste do Caddy para encaminhar `/developers`, `/v3/api-docs` e assets do Swagger para o core.

## Endpoints No Grupo Público

O grupo `rag-core` inclui:

- `POST /api/chat`;
- `GET/POST /api/documents`;
- `GET /api/workspaces`;
- conversas;
- observabilidade;
- handoffs;
- ferramentas controladas;
- versão.

Ficam fora do grupo:

- endpoints administrativos;
- `site-chat` interno;
- webhooks de canal.

## Segurança

- O console não contém API key real.
- O usuário precisa informar uma API key em `Authorize`.
- Endpoints protegidos continuam exigindo `X-OPIAGILE-API-KEY` quando `API_SECURITY_REQUIRE_API_KEY=true`.
- A API key deve ser usada server-to-server, não em frontend público.
- A documentação interativa pode ser desligada:

```text
OPENAPI_ENABLED=false
OPENAPI_UI_ENABLED=false
```
