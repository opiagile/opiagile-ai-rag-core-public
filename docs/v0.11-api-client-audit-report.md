# Relatório v0.11 — Auditoria Operacional De API Clients

## Resumo Executivo

A v0.11 adicionou auditoria operacional para API keys tenant-aware. O core agora registra uso permitido e bloqueado sem gravar API keys, payloads ou segredos.

Essa camada ajuda a acompanhar consumo por gateway, erros de escopo, rate limit e últimos acessos antes de abrir mais tráfego ou novos canais.

## O Que Foi Implementado

- Migration `V13__api_client_usage_logs.sql`.
- Tabela `api_client_usage_logs`.
- Registro de eventos no `ApiClientAuthenticationFilter`.
- Endpoint administrativo:

```text
GET /api/admin/api-clients/usage
```

- Resumo por API client.
- Lista de eventos recentes.
- Proteção por `DEMO_ADMIN_TOKEN`.

## Campos Auditados

- API client;
- tenant/workspace;
- método HTTP;
- path;
- escopo exigido;
- status HTTP;
- permitido/bloqueado;
- motivo de bloqueio;
- IP;
- user agent;
- latência;
- data/hora.

Não são registrados:

- API key;
- payload da mensagem;
- documentos enviados;
- tokens externos;
- segredos de LLM, WhatsApp ou Langfuse.

## Exemplos De Uso

```bash
curl http://localhost:8080/api/admin/api-clients/usage \
  -H "X-Demo-Admin-Token: $DEMO_ADMIN_TOKEN"
```

Com limite:

```bash
curl "http://localhost:8080/api/admin/api-clients/usage?limit=100" \
  -H "X-Demo-Admin-Token: $DEMO_ADMIN_TOKEN"
```

## Limitações

- O rate limit segue em memória.
- A auditoria é best effort: falha de log não derruba a requisição principal.
- O endpoint administrativo ainda usa token simples legado. Um painel futuro deve usar OIDC/JWT e RBAC.
