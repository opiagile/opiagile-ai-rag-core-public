# Relatório v0.15 — Java 25 LTS, Spring Boot 4.1 E Spring AI 2.0

Data da sprint: 2026-06-30

## Resumo Executivo

A v0.15 atualiza o baseline técnico do `opiagile-ai-rag-core` para Java 25 LTS, mantendo o escopo do repositório como core RAG sem frontend.

A atualização foi feita de forma conservadora:

- Java 25 LTS como baseline de compilação e runtime.
- Spring Boot 4.1.0.
- Spring AI 2.0.0.
- Springdoc OpenAPI 3.0.3.
- Maven Wrapper 3.9.12.
- Imagens Docker com Eclipse Temurin 25 para build e runtime.
- Workflows GitHub Actions ajustados para Java 25.
- Deploy Oracle mantido na branch `develop`.
- Imports e tratamento de exceções migrados para Jackson 3, exigido pela linha Spring Boot 4.
- Auto-configuração Flyway restaurada com o módulo `spring-boot-flyway`, exigido pela modularização do Spring Boot 4.

Java 26 não foi considerado nesta sprint.

## Objetivo

Modernizar a base Java/Spring para a linha LTS atual escolhida pelo projeto, preservando:

- modo local sem chave externa;
- RAG com fontes;
- fallback textual;
- embeddings OpenAI opcionais;
- integração Spring AI para embeddings;
- deploy Oracle Cloud Always Free;
- compatibilidade ARM64.

## Decisões Técnicas

- O `pom.xml` passou a usar `spring-boot-starter-parent` `4.1.0`.
- A propriedade `java.version` passou para `25`.
- O BOM Spring AI passou para `2.0.0`.
- O Springdoc OpenAPI foi atualizado para `3.0.3`.
- O Maven Wrapper passou para Maven `3.9.12`.
- O workflow de deploy Oracle valida e publica a branch `develop`, mantendo a VPS como ambiente de demonstração controlado.
- O Dockerfile Oracle Free Tier passou a usar:
  - `maven:3.9.12-eclipse-temurin-25`;
  - `eclipse-temurin:25-jre-noble`.
- A validação local usou Docker porque a máquina host ainda tem Java 21 instalado.
- O Spring Boot 4 usa Jackson 3 por padrão, com pacotes `tools.jackson.*`.
- O código que dependia de `com.fasterxml.jackson.*` foi migrado para Jackson 3 para manter serialização, filtros de erro, providers OpenAI, parser WhatsApp e testes alinhados ao runtime.
- Os primeiros workflows após o commit inicial da v0.15 falharam no startup por ausência de bean `com.fasterxml.jackson.databind.ObjectMapper`; a causa foi corrigida migrando para `tools.jackson.databind.ObjectMapper`.
- O Spring Boot 4 também separa a integração Flyway no artefato `spring-boot-flyway`. Sem esse módulo, a aplicação subia, mas as migrations não eram executadas no CI e endpoints que dependem de `tenants` falhavam com HTTP 500.

Observação: durante os testes com Java 25, o Mockito exibiu aviso sobre carregamento dinâmico de agente Java. Isso não quebrou a suíte, mas deve ser acompanhado em versões futuras do JDK.

Observação adicional: a primeira execução dos workflows `validar-llm-real` e `validar-spring-ai-embeddings` após o commit inicial falhou antes dos testes funcionais, durante o startup da API. O erro apontava para injeção de `ObjectMapper` do Jackson 2. A correção complementar desta sprint migrou o projeto para Jackson 3.

Observação complementar: após corrigir Jackson 3, os workflows passaram pelos testes e pelo startup, mas falharam no upload do documento porque as migrations Flyway não haviam rodado no ambiente GitHub Actions. A causa foi ausência do módulo `spring-boot-flyway`, agora adicionado ao `pom.xml`. Após essa correção, os dois workflows críticos passaram em Java 25.

## Limitações

- A máquina local do agente ainda possui Java 21 como JDK instalado no host.
- A validação Java 25 foi feita via imagem Docker oficial.
- A validação real com `OPENAI_API_KEY` deve continuar sendo executada pelo workflow dedicado, usando secret do GitHub.
- Esta sprint não altera comportamento funcional de RAG, chat, embeddings, API keys, OpenAPI ou deploy.
