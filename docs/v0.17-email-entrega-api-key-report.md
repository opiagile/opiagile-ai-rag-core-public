# Relatório v0.17 - Email De Entrega De API Key Sandbox

## Resumo Executivo

A v0.17 completa o fluxo de solicitação e aprovação de API key sandbox com envio assíncrono de emails transacionais e link de uso único para o lead acessar a chave.

O formulário do portal Developers continua salvando a solicitação primeiro no banco. A aprovação administrativa continua protegida por `X-Demo-Admin-Token`, mas agora cria também uma entrega segura da chave:

- confirmação para o lead;
- notificação interna para `contato@opiagile.com`;
- email de aprovação com link de uso único;
- cópia operacional para `contato@opiagile.com`;
- página HTML com layout Opiagile para revelar a API key uma única vez.

O único email visível para leads e usuários é `contato@opiagile.com`.

## O Que Foi Implementado

- Tabela `developer_access_email_outbox` para fila assíncrona de emails.
- Tabela `developer_access_key_deliveries` para entrega temporária da chave por token.
- Templates HTML/texto para emails de solicitação recebida, nova solicitação e sandbox aprovado.
- Scheduler de email baseado em outbox, com retry, tentativas e marcação de falha.
- Link `/developers/access-key/{token}` para revelar a API key uma única vez.
- Criptografia temporária da API key na entrega, sem persistir a chave em texto claro.
- Configurações para URL pública e validade do link.
- Documentação do fluxo no README, exemplos de API e painel interno.

## Segurança

- API key operacional continua armazenada somente como prefixo e hash em `api_clients`.
- A chave completa é exibida apenas uma vez ao admin e uma vez ao lead pelo link.
- O token do link é armazenado como hash.
- A entrega expira por configuração e é invalidada no primeiro uso.
- Emails são processados de forma assíncrona; falha SMTP não quebra o formulário nem a aprovação.
- Nenhuma senha, token SMTP ou API key real foi adicionada ao Git.

## Variáveis De Ambiente

```text
DEVELOPER_ACCESS_EMAIL_ENABLED=true
DEVELOPER_ACCESS_EMAIL_TO=contato@opiagile.com
DEVELOPER_ACCESS_EMAIL_FROM=contato@opiagile.com
DEVELOPER_ACCESS_EMAIL_PUBLIC_BASE_URL=https://opiagile.com
DEVELOPER_ACCESS_KEY_DELIVERY_EXPIRES_IN_HOURS=24
DEVELOPER_ACCESS_EMAIL_SCHEDULER_ENABLED=true
DEVELOPER_ACCESS_EMAIL_SCHEDULER_FIXED_DELAY_MS=60000
DEVELOPER_ACCESS_EMAIL_SCHEDULER_BATCH_SIZE=5
DEVELOPER_ACCESS_EMAIL_SCHEDULER_MAX_ATTEMPTS=10
DEVELOPER_ACCESS_EMAIL_SCHEDULER_RETRY_DELAY_MINUTES=15
```

Para Microsoft 365, o usuário autenticado no SMTP precisa ter permissão para enviar como `contato@opiagile.com` quando esse endereço for alias, caixa compartilhada ou mailbox separada.

## Limitações

- O envio real depende de SMTP configurado fora do Git.
- O remetente `contato@opiagile.com` depende de permissão `Send as`/equivalente no Microsoft 365.
- Ainda não há painel visual para reenviar email de entrega.
- Ainda não há tela administrativa para revogar API keys.

## Próximos Passos

1. Configurar `DEVELOPER_ACCESS_EMAIL_FROM=contato@opiagile.com` na VPS somente após validar permissão de envio.
2. Aprovar uma solicitação real e confirmar recebimento do link pelo lead.
3. Adicionar ação administrativa de reenvio seguro se o link expirar sem uso.
4. Criar revogação visual de API keys no painel interno.
