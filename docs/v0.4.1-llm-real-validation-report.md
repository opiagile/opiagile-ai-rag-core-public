## Validação Real Com OpenAI

Como o `.env` local não continha chave OpenAI, foi criado um workflow manual seguro:

- `.github/workflows/validar-llm-real.yml`
- `scripts/validate-llm-real.py`

O workflow sobe PostgreSQL/pgvector, executa testes do backend, inicia a API com `CHAT_RESPONSE_MODE=LLM`, `LLM_PROVIDER=OPENAI` e `OPENAI_API_KEY` vindo de GitHub Secret, ingere `samples/clinica/faq.txt` e executa o conjunto de perguntas.

## Ajustes Feitos Durante A Sprint

A primeira validação real mostrou que algumas respostas caíam em fallback porque a chamada retornava sem texto extraível. Foram feitos ajustes pontuais:

- `OpenAiChatModelProvider` passou a enviar `reasoning.effort=minimal`.
- `OpenAiChatModelProvider` passou a enviar `text.verbosity=low`.
- O erro de extração de texto agora inclui status e motivo de incompletude, sem expor segredo.
- O workflow passou a usar `set -o pipefail` para falhar corretamente quando o script falhar.
- A regra local classificou `cirurgia`, `emergência`, `urgência` e `pronto socorro` como `FORA_DO_ESCOPO` para evitar resposta indevida em contexto de clínica demo.
- O script de validação passou a diferenciar fallback operacional (`USUARIO_PEDIU_HUMANO`, `FORA_DO_ESCOPO`) de falha do provider OpenAI.

## Limitações Mantidas

- Embeddings reais ainda não são o fluxo principal.
- Recuperação segue textual/local antes da geração LLM.
- WhatsApp real ainda não foi implementado.
- Demo pública HTTPS ainda está pendente.
- Autenticação, rate limit e proteção de demo pública continuam pendentes.
- Uso de LLM depende de chave, disponibilidade de modelo e custo do provedor.

## Recomendação

A v0.4.1 validou a experiência conversacional real. O projeto está apto a seguir para:

> v0.5 - WhatsApp Cloud API Piloto Controlado

A v0.5 deve manter allowlist, dry-run por padrão, webhook HTTPS, validação de assinatura e envio real apenas para números autorizados.
