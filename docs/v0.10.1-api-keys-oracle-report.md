# Relatório v0.10.1 — API Keys No Ambiente Oracle

## Resumo Executivo

A v0.10.1 preparou o deploy Oracle para operar com API keys tenant-aware entre o core RAG e aplicações consumidoras, sem expor chaves ao navegador.

O modo legado segue disponível enquanto as chaves são cadastradas e validadas. A ativação estrita deve ocorrer somente depois que landing e demo web estiverem usando as novas chaves.

## O Que Foi Ajustado

- `deploy/oracle-free-tier/docker-compose.yml` agora repassa variáveis de API security para o container do core.
- O serviço `site` recebe `OPIAGILE_RAG_CORE_API_KEY`.
- O serviço `demo-web` recebe uma chave geral e chaves específicas por workspace.
- `.env.production.example` documenta as variáveis novas.
- O contrato recomendado é `X-OPIAGILE-API-KEY` server-to-server.
- `scripts/configure-oracle-api-clients.py` automatiza o cadastro seguro dos clients na VPS Oracle, sem imprimir API keys.

## Estratégia De Chaves

Landing Opiagile:

- tenant: `opiagile`;
- workspace: `opiagile-rag`;
- escopos recomendados: `chat:write`.

Demo RAG:

- tenant: `demo`;
- workspaces:
  - `clinica-demo`;
  - `atendimento-demo`;
  - `locacao-demo`;
- escopos recomendados:
  - `chat:write`;
  - `documents:read`;
  - `documents:upload`;
  - `conversations:read`;
  - `observability:read`;
  - `workspaces:read`.

## Sequência Operacional Recomendada

1. Deployar o core com migrations v12 e v13.
2. Gerar e cadastrar API keys com `scripts/generate-api-client-key.sh` ou `scripts/configure-oracle-api-clients.py` no ambiente Oracle.
3. Preencher `.env` do Oracle sem imprimir valores reais.
4. Reiniciar `api`, `site` e `demo-web`.
5. Validar com `API_SECURITY_REQUIRE_API_KEY=false`.
6. Confirmar que site e demo já enviam `X-OPIAGILE-API-KEY`.
7. Trocar `API_SECURITY_REQUIRE_API_KEY=true`.
8. Validar que chamadas sem API key retornam `401`.

## Resultado No Oracle

O ambiente Oracle foi configurado com API keys tenant-aware para:

- landing Opiagile;
- Demo RAG clínica;
- Demo RAG atendimento;
- Demo RAG locação.

As chaves foram gravadas apenas no `.env` da VPS. O banco recebeu somente prefixo e hash SHA-256.

O modo estrito foi ativado:

```text
API_SECURITY_ENABLED=true
API_SECURITY_REQUIRE_API_KEY=true
```

Validações operacionais executadas:

- chamada sem API key para endpoint protegido retornou `401`;
- chamada com API key da Demo RAG retornou `200`;
- chat da landing pública retornou `200`;
- relatório administrativo de uso retornou `200` com `DEMO_ADMIN_TOKEN`.

## Variáveis Novas No Oracle

```text
API_SECURITY_ENABLED=true
API_SECURITY_REQUIRE_API_KEY=false
API_SECURITY_DEFAULT_RATE_LIMIT_PER_MINUTE=60
OPIAGILE_SITE_RAG_CORE_API_KEY=
DEMO_RAG_CORE_API_KEY=
DEMO_RAG_CORE_API_KEY_CLINICA_DEMO=
DEMO_RAG_CORE_API_KEY_ATENDIMENTO_DEMO=
DEMO_RAG_CORE_API_KEY_LOCACAO_DEMO=
```

## Segurança

- API keys reais devem ficar apenas no `.env` da VPS ou em secret manager.
- A tabela `api_clients` armazena apenas hash SHA-256.
- O navegador não recebe a API key.
- Headers `X-Tenant-Id` e `X-Workspace-Id` continuam aceitos somente para compatibilidade quando o modo estrito estiver desligado.
