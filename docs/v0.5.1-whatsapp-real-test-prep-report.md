A v0.5.1 prepara o projeto para o primeiro teste real com WhatsApp Cloud API direta da Meta, sem adicionar features grandes e sem tratar o piloto como produção.

## Smoke Tests Locais

Executados com PostgreSQL local e API Spring Boot em `localhost:8080`:

- `GET /actuator/health`: retornou `UP`.
- `GET /api/webhooks/whatsapp/status`: retornou `provider=MOCK`, `dryRun=true`, `sendEnabled=false`, `signatureRequired=true`, campos sensíveis apenas como booleanos e `allowedTestNumbersCount=0`. Nenhum token, app secret, verify token ou telefone completo foi exposto.
- `GET /api/webhooks/whatsapp/meta` com `TOKEN_DEMO`: retornou HTTP 403, esperado porque o `.env` local não possui `WHATSAPP_VERIFY_TOKEN` configurado.
- `scripts/check-whatsapp-pilot.sh`: executou com parser seguro do `.env`, sem imprimir segredos.

Durante o smoke test, a primeira tentativa de subir a API revelou um problema de runtime no bean `WhatsAppRateLimiter`: havia dois construtores e o Spring não conseguia escolher o construtor com `WhatsAppProperties`. A correção mínima foi marcar o construtor público com `@Autowired`. Após a correção, a API subiu e os smoke tests passaram.

## Critérios Para Liberar Primeiro Envio Real

- Dry-run recebeu payload real da Meta.
- Assinatura foi validada.
- Número do responsável está na allowlist.
- `ChatService` respondeu corretamente.
- `GET /api/webhooks/whatsapp/status` não expõe segredos.
- Logs não exibem token, app secret, verify token ou telefone completo.
- `WHATSAPP_SEND_ENABLED=true` e `WHATSAPP_DRY_RUN=false` só são aplicados depois do teste dry-run.

## Rollback

- `WHATSAPP_SEND_ENABLED=false`
- `WHATSAPP_DRY_RUN=true`
- Remover número da allowlist.
- Desligar túnel ou URL pública.
- Revogar token se houver suspeita de exposição.
