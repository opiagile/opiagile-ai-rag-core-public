# Relatório v0.16.2 — Sandbox Temporário Para Leads

Data da sprint: 2026-06-30

## Resumo Executivo

A v0.16.2 adiciona um fluxo operacional para aprovar solicitações do portal developers criando tenant/workspace temporário com o nome do cliente.

Esse fluxo não cria acesso automaticamente para qualquer visitante. A criação continua sendo uma ação administrativa, protegida por `X-Demo-Admin-Token`, dentro do painel interno acessado por túnel SSH.

## O Que Mudou

- API keys podem ter `expires_at`.
- Tenants e workspaces podem ser marcados como `sandbox=true` com `expires_at`.
- O painel interno passa a ter dois modos:
  - criar sandbox temporário para o lead;
  - usar workspace existente.
- O modo temporário cria tenant/workspace do cliente e retorna `expiresAt`.
- Um scheduler remove sandboxes temporários expirados.
- A página `/developers` informa que o sandbox é temporário e explica a retenção.

## Retenção E LGPD

Ao expirar, o tenant/workspace temporário é excluído do sandbox, junto com documentos, chunks, conversas e logs de recuperação daquele ambiente.

A solicitação do lead permanece em `developer_access_requests` para contato, auditoria operacional e continuidade comercial. Essa separação é intencional para respeitar LGPD: dados enviados ao sandbox de teste não ficam indefinidamente disponíveis, enquanto o registro comercial mínimo segue preservado.

## Endpoint Administrativo

```text
POST /api/admin/developer-access-requests/{requestId}/approve-temporary-sandbox
```

Payload:

```json
{
  "scopes": ["chat:write", "documents:upload", "documents:read", "providers:read", "workspaces:read"],
  "rateLimitPerMinute": 30,
  "clientName": "Sandbox - Empresa Demo",
  "expiresInHours": 24
}
```

## Limpeza Automática

Configurações:

```text
DEVELOPER_ACCESS_SANDBOX_CLEANUP_ENABLED=true
DEVELOPER_ACCESS_SANDBOX_CLEANUP_FIXED_DELAY_MS=300000
DEVELOPER_ACCESS_SANDBOX_CLEANUP_BATCH_SIZE=25
```

O cleanup só remove tenants marcados com `sandbox=true` e `expires_at <= now()`.
