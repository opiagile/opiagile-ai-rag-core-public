# Relatório v0.14 — Code Review Spring AI E Ecossistema Spring

## Resumo Executivo

Esta v0.14 é uma análise técnica do código atual, não uma refatoração. O objetivo é deixar evidente onde o core já usa bem o ecossistema Spring e onde ainda existe implementação manual que deve ser reavaliada antes das próximas evoluções.

Conclusão principal:

- O projeto usa bem Spring Boot, Spring Web, Validation, JDBC, Flyway, Actuator, Mail, Scheduler e springdoc.
- O projeto ainda não usa Spring AI diretamente para chat, embeddings, vector store, advisors ou tool calling.
- A integração OpenAI está funcional, testada e isolada em providers próprios, mas usa `RestClient` manual.
- A recuperação pgvector está funcional, mas usa SQL manual com `JdbcClient`.
- A próxima sprint técnica deve avaliar migração gradual para Spring AI sem quebrar o contrato atual de API, fallback local e observabilidade.

## Pontos Fortes Encontrados

### 1. Arquitetura Com Providers Próprios

`ChatModelProvider` e `EmbeddingProvider` já isolam dependência externa. Isso reduz risco de lock-in e facilita a futura entrada de Spring AI, Ollama, provider OpenAI-compatible ou fallback local.

Essa decisão deve ser preservada. A migração ideal não deve espalhar `ChatClient`, `EmbeddingModel` ou classes de provider diretamente em `ChatService` e `DocumentService`; o contrato interno do core deve continuar simples.

### 2. Fallback Local Continua Bem Separado

O modo sem chave externa ainda funciona por:

- `DemoChatModelProvider`;
- `NoopEmbeddingProvider`;
- fallback textual local.

Essa característica é importante para portfólio, testes locais e continuidade operacional.

### 3. pgvector Está Implementado Com Controle Explícito

`RagChunkRepository.findNearestIndexedChunks(...)` usa operador `<=>` do pgvector e respeita tenant/workspace. A implementação é direta, compreensível e testável.

Antes de trocar por `PgVectorStore`, é preciso garantir que:

- o filtro por tenant/workspace continue obrigatório;
- os metadados de documento, arquivo, idioma e chunk continuem retornando no formato atual;
- o fallback textual continue disponível;
- os logs de retrieval continuem auditáveis.

### 4. Ferramentas Controladas Foram Desenhadas Com Defesa

O módulo de tools não deixa a LLM executar SQL livre. O planner escolhe ações permitidas, e o backend decide consultas pré-aprovadas. Isso é uma boa decisão de segurança.

Spring AI Tool Calling pode ser útil no futuro, mas não deve substituir essa camada por execução livre de ferramentas. O modelo deve continuar sem credenciais e sem poder montar SQL arbitrário.

## Principais Achados

### Achado 1 — Dependência Spring AI Não Está Presente

`backend/pom.xml` não possui dependências `org.springframework.ai`.

Impacto:

- O projeto menciona Spring AI como alvo de stack, mas o fluxo real de chat e embeddings usa HTTP manual.
- A manutenção dos contratos OpenAI fica sob responsabilidade do projeto.
- Recursos prontos de `ChatClient`, `EmbeddingModel`, `VectorStore`, advisors e tool calling não são aproveitados.

Recomendação:

- Criar sprint curta para introduzir Spring AI atrás das interfaces existentes.
- Começar por embeddings, porque o contrato `EmbeddingProvider` é pequeno e o risco é menor.
- Depois avaliar chat e ferramentas.

### Achado 2 — Chat OpenAI Usa Responses API Manual

`OpenAiChatModelProvider` monta payload da OpenAI manualmente com `RestClient`.

Pontos positivos:

- Funciona com a Responses API.
- Permite controlar `reasoning.effort`, `text.verbosity`, timeout e extração de tokens.
- Está isolado em um provider.

Riscos:

- Mudanças de contrato da OpenAI exigem manutenção manual.
- A extração de texto e tokens depende da estrutura JSON atual.
- Não aproveita abstrações Spring AI para chat, advisors, memory ou tool calling.

Recomendação:

- Manter esse provider até haver teste de regressão.
- Criar `SpringAiChatModelProvider` como implementação alternativa.
- Usar feature flag para comparar `OPENAI_REST_MANUAL` versus `SPRING_AI`.

### Achado 3 — Embeddings OpenAI Também Usam HTTP Manual

`OpenAiEmbeddingProvider` chama `/embeddings` com `RestClient`, converte JSON para `float[]` e silencia falhas retornando `Optional.empty()`.

Pontos positivos:

- Fallback é seguro e não derruba a ingestão.
- O provider é pequeno.

Riscos:

- Não usa `EmbeddingModel`.
- Erros podem ficar invisíveis demais se não houver métrica/log operacional.
- Dimensão do vetor precisa continuar alinhada ao schema `vector(1536)`.

Recomendação:

- Primeira migração candidata: `SpringAiEmbeddingProvider`.
- Validar dimensão, custo, timeout, fallback e logs.
- Preservar `NoopEmbeddingProvider`.

### Achado 4 — Vector Store Manual Pode Ser Mantido, Mas Deve Ser Reavaliado

O uso manual de SQL com pgvector não é necessariamente um problema. Para multi-tenant/workspace, controle de metadados e logging próprio, SQL explícito pode ser mais claro do que uma abstração genérica.

Recomendação:

- Avaliar `PgVectorStore` em uma prova de conceito isolada.
- Não migrar enquanto não houver garantia de filtro tenant/workspace obrigatório.
- Se a abstração aumentar complexidade, manter `JdbcClient` manual neste ponto.

### Achado 5 — Prompt Builder Manual É Aceitável, Mas Poderia Evoluir Para Advisors

`RagAnswerPromptBuilder` centraliza instruções, histórico, fontes, idioma, ferramentas e resultados. Isso evita prompt em controller, o que é correto.

Pontos de atenção:

- O prompt cresce a cada sprint.
- Regras de idioma, fontes, handoff, demo e tools estão no mesmo template.
- Spring AI Advisors poderiam separar responsabilidades, mas a migração precisa preservar comportamento atual.

Recomendação:

- Antes de migrar, criar testes de snapshot/contrato para o prompt.
- Separar blocos internos por responsabilidade mesmo sem Spring AI.
- Avaliar advisors somente depois de estabilizar testes de qualidade.

### Achado 6 — Rate Limit Em Memória É Simples E Limitado

`SimpleRateLimiter` resolve o cenário atual, mas é local ao processo.

Impacto:

- Em uma instância única, atende o piloto.
- Em múltiplas réplicas, o limite não é global.
- Em reinício, o estado zera.

Recomendação:

- Manter por enquanto.
- Se houver exposição pública relevante, avaliar Redis, Bucket4j ou Resilience4j.

### Achado 7 — Segurança Com Filtros Próprios É Suficiente Para API Key, Mas Pode Evoluir

`ApiClientAuthenticationFilter` implementa API key tenant-aware, escopos, rate limit e auditoria.

Pontos positivos:

- Comportamento explícito e testado.
- Bom encaixe para server-to-server.

Limite:

- Não é autenticação de usuário final.
- Não há RBAC/OIDC para painel administrativo.

Recomendação:

- Manter API keys para gateways.
- Se houver painel multiusuário, adicionar Spring Security com OIDC/JWT e papéis.

## Plano Recomendado

### v0.13 — Resiliência De Provedores IA

Objetivo:

- Mapear e expor status seguro dos providers.
- Preparar configuração de provider/fallback sem trocar implementação.
- Melhorar visibilidade operacional de LLM, embeddings e fallback.

Entregáveis:

- Endpoint seguro de status de providers.
- Configuração documentada de prioridade/fallback.
- Relatório de continuidade operacional.
- Testes de fallback.

### v0.14 — Code Review Spring AI E Refatoração Guiada

Objetivo:

- Executar uma prova de conceito de Spring AI sem quebrar providers atuais.
- Decidir o que deve migrar e o que deve permanecer manual.

Ordem recomendada:

1. Introduzir dependências Spring AI compatíveis com Spring Boot/Spring AI alvo do projeto.
2. Criar `SpringAiEmbeddingProvider` atrás de `EmbeddingProvider`.
3. Validar ingestão com embeddings e recuperação pgvector.
4. Criar `SpringAiChatModelProvider` atrás de `ChatModelProvider`.
5. Comparar qualidade, tokens, latência e fallback contra o provider atual.
6. Avaliar `PgVectorStore` separadamente.
7. Avaliar Spring AI tool calling sem relaxar a política de SQL read-only.

Critério de aceite:

- Nenhuma regressão em `./mvnw test`.
- API pública sem quebra.
- Fallback DEMO preservado.
- Tenant/workspace preservados em todo fluxo.
- Fontes e trace preservados.
- Relatório comparando manual versus Spring AI.

## Decisão Recomendada Agora

Não substituir tudo por Spring AI de uma vez.

O caminho mais seguro é incremental:

1. Documentar este review como baseline.
2. Fazer v0.13 de resiliência/status.
3. Fazer v0.14 com prova de conceito de Spring AI em embeddings.
4. Migrar chat somente se a comparação mostrar ganho real de manutenção sem perda de controle.
